# Requirements and User Stories

Software engineering starts by understanding the problem before writing code.

A requirement should describe what the system must do or what constraint it must satisfy. A user story expresses a requirement from the user's perspective.

A useful pattern:

> As a [type of user], I want [goal], so that [reason].

Good requirements are specific, testable, and scoped. Bad requirements are vague, emotional, or too broad, such as "make it better" or "build an app like YouTube."

For a small project, define:
- The target user
- The main problem
- Core features
- Non-goals
- Acceptance criteria
